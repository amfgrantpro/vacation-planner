## 1. Sprint 3 in One Paragraph

Sprint 3 replaced the open-ended conversational agent from Sprint 2 with a **structured decision consultant**. The agent now guides users through a rigid 4-phase funnel (Intake → Explore → Shortlist → Compare) using an explicit, persistent "cart" of destination candidates. Three iterative builds were required: Build 1 laid the architecture but broke on API parsing; Build 2 fixed parsing but broke the agent's conversational behavior; Build 3 restored the correct industry-standard agentic loop while keeping Build 2's improvements. The result is a working end-to-end agent that moves users through the decision funnel, manages a candidate cart, and generates a structured comparison matrix.

## 3. Why Sprint 3 Required Three Builds

Sprint 3 was planned as a single build. It required three because each build exposed a different category of failure that could not have been discovered without running the system.

### Build 1: The Foundation (sprint-3-build.md)

**What was built**: The full architecture as specced — 4-phase state machine, `DestinationCandidate` model, 4 tools, `ActiveCartSidebar`, `ComparisonMatrix`, structured JSON response.

**What failed**: The `update_plan` tool schema was auto-generated via `VacationPlanPatch.model_json_schema()`. Pydantic's output contains nested `$defs` and `$ref` references. The **Groq API parser rejected the tool definition outright** with `invalid_request_error` before the LLM ever saw it. State mutations (adding candidates, transitioning phases) never executed.

**Root cause**: We trusted Pydantic's schema generation without verifying it against the Groq API's tool schema requirements.

**What was kept**: Everything else — the state machine, the candidate model, the frontend, the 4-tool surface, the structured JSON response contract. None of this was the problem.

---

### Build 2: The Fix That Broke Behavior (sprint-3-fix.md)

**What was fixed**: Tool schemas were rewritten as hand-crafted flat JSON (all primitive types, no `$defs`). This resolved the parser rejection.

**What was also changed (and shouldn't have been)**: The dual-call architecture was replaced with a **single-call pattern**, based on general advice that "dual-prompts are an anti-pattern." The second LLM call was removed. Tool results were no longer fed back to the LLM before responding.

**What failed**: The agent executed tools but never observed the results before generating its reply. Every turn produced responses like "I have updated the plan accordingly" with no follow-up question. The user had to re-prompt manually to move the conversation forward. **This is not an agentic behavior — it is a broken chatbot.**

**Why the advice was wrong**: "Single-call is better" applies to **simple chatbots** (no tools). For **agentic systems**, the LLM must observe tool results before responding — this is the definition of a ReAct loop. Removing the second call violated the project brief's explicit requirement for production-grade agentic patterns.

**What was kept**: Flat tool schemas, outcome-focused prompt language, phase-gated tool visibility, fail-fast error handling.

---

### Build 3: The Correct Architecture (sprint-3-fix2.md)

**What was fixed**: The dual-call pattern was restored as a **conditional ReAct loop**:
- If the LLM calls tools → execute tools → feed results back → Call 2 with `tool_choice="none"` → LLM observes and responds.
- If no tools are called → use the original response directly (no wasted second call).

**Result**: The agent now behaves consultatively in a single user turn. Tool calls execute, the LLM observes updated state, and it generates a contextually informed response with a follow-up question.

**Verification**: A full conversation from Intake → Explore → Shortlist → Compare completed in 8 POST /chat requests with zero tool call failures and zero extra re-prompts required from the user.

---

### The Lesson for Sprint 4

> **Never revert to Build 1 or Build 2 patterns.**
>
> - **Build 1 mistake**: Do not auto-generate Groq tool schemas from Pydantic models. Always write flat, hand-crafted JSON tool schemas.
> - **Build 2 mistake**: Do not remove the second LLM call in an agentic system. The ReAct loop (Reason → Act → Observe → Respond) is not an anti-pattern here — it is a requirement. Single-call is only correct for chatbots with no tools.

### 4.1 The 4-Phase Gated State Machine

The agent operates in one of four phases, stored as a `Phase` enum on `VacationPlan`. Each phase has a distinct goal enforced by phase-specific prompt instructions:

| Phase | Goal | Entry Guardrail | Exit Condition |
|---|---|---|---|
| **INTAKE** | Profile the user: origin, duration, travelers, vacation purpose, travel history | Starting state | Agent has origin, duration, travelers, and travel style → transitions to EXPLORE |
| **EXPLORE** | Diagnose preferences; suggest destinations one at a time; curate top 3 | None | 3 solid active candidates the user is genuinely interested in → transitions to SHORTLIST |
| **SHORTLIST** | Get formal user confirmation on the top 3 | None | User confirms (or one round of refinement) → transitions to COMPARE |
| **COMPARE** | Generate MCDM matrix; present trade-offs; let user decide | **≥ 2 active candidates required** (enforced server-side) | User picks a winner or requests to go back |

Backward movement is explicitly allowed. The user can always return to an earlier phase (e.g., Compare → Explore to add a new candidate).

Phase transitions are:
1. Requested by the LLM via the `transition_phase` tool
2. **Validated server-side** in `_can_transition()` before being applied
3. The only mechanism for changing `plan.phase`; the LLM cannot write to it directly

---

### 4.2 Flat JSON Tool Schemas (Build 2 Refinement)

All four tool schemas are hand-written flat JSON. No Pydantic schema generation, no `$defs`, no `$ref`. This is a permanent constraint — the Groq API parser cannot handle nested schema references.

```
TOOL_UPDATE_PLAN         → All fields as scalar parameters (string, integer, array of strings)
TOOL_MANAGE_CANDIDATES   → action (enum) + candidates array (inline object schema, no $ref)
TOOL_TRANSITION_PHASE    → target_phase (enum string)
TOOL_GENERATE_MCDM       → criteria (array of objects)
```

#### Phase-Filtered Tool Availability

The orchestrator only exposes tools valid for the current phase:

| Phase | Tools Available |
|---|---|
| INTAKE | `update_plan`, `manage_candidates`, `transition_phase` |
| EXPLORE | `update_plan`, `manage_candidates`, `transition_phase` |
| SHORTLIST | `update_plan`, `manage_candidates`, `transition_phase` |
| COMPARE | `update_plan`, `transition_phase`, `generate_mcdm_matrix` |

`manage_candidates` is not exposed in COMPARE (the shortlist is locked). `generate_mcdm_matrix` is not exposed in earlier phases.

---

### 4.3 The Conditional Dual-Call ReAct Loop

This is the engine running the system. It is implemented in `AgentOrchestrator.run_turn()`.

```
User Input arrives at POST /chat
        │
        ▼
Build system prompt with current plan state + phase-specific instructions
        │
        ▼
LLM Call 1  [tool_choice="auto", phase-filtered tools]
        │
        ├── No tool calls?
        │       └── Use Call 1's text directly → return response
        │
        └── Tool calls present?
                │
                ▼
        Execute each tool call in sequence
        → Apply mutations to VacationPlan
        → Record tool results as {"role": "tool"} messages
                │
                ▼
        Rebuild system prompt with UPDATED plan state
                │
                ▼
        LLM Call 2  [tool_choice="none", same phase-filtered tools]
        → LLM observes: updated state + tool result summaries
        → Generates contextually informed response
                │
                ▼
Return: { text_reply, comparison_matrix } + updated_plan + new_messages
```

**Why "conditional"**: If the LLM does not call any tools (e.g., a simple conversational reply), no second call is made. This saves tokens on lightweight turns while preserving full ReAct behavior when tool use occurs.

**Why this is the industry standard**: LangGraph, Anthropic's tool use API, and OpenAI Assistants all implement this pattern. The LLM must observe tool results before responding — otherwise it cannot reason about what just changed. A single call where tools execute but results are unseen is not an agentic pattern; it is a broken chatbot.

### 4.4 State Schema

**`models.py`** — final verified state:

```python
class Phase(str, Enum):
    INTAKE = "intake"
    EXPLORE = "explore"
    SHORTLIST = "shortlist"
    COMPARE = "compare"

class TripShape(BaseModel):           # Sprint 2 — unchanged
    origin: Optional[str]
    duration_days: Optional[int]
    travelers: int = 1
    pax_description: str = ""

class MentalModel(BaseModel):         # Sprint 2 — unchanged
    knowns: List[str] = []
    unknowns: List[str] = []          # Systemic decision blockers, NOT last question asked
    sentiments: List[str] = []

class DestinationCandidate(BaseModel): # NEW in Sprint 3
    name: str
    status: str = "active"            # "active" or "eliminated"
    rationale: str = ""               # Why kept or removed
    pros_cons: Optional[dict] = None  # Populated in Compare phase
    decision_criteria: Optional[dict] = None  # Unresolved blockers per destination

class VacationPlan(BaseModel):
    phase: Phase = Phase.INTAKE       # NEW in Sprint 3
    vacation_purpose: str = ""
    trip_shape: TripShape
    mental_model: MentalModel
    candidates: List[DestinationCandidate] = []  # NEW in Sprint 3 — the "Cart"
    budget_range: Optional[str] = None
    comparison_matrix: Optional[List[dict]] = None  # NEW in Sprint 3 — MCDM rows
    notes: str = ""
```

**Candidate upsert rule**: Candidates are matched by `name.lower()` and upserted — never replaced as a full array. The LLM cannot accidentally wipe the list with a partial update.

**Elimination rule**: Eliminated candidates are retained in the array with `status="eliminated"` and a recorded rationale. They vanish from the UI but persist in state for decision auditability.

### 4.5 Prompting System

**`prompt.py`** uses outcome-focused language throughout. Prompts describe *what the agent should achieve*, not *how to call functions*.

Structure per turn:
```
TEMPLATE header
  → "You are an expert Travel Consultant..."
  → "IMPORTANT: Simply respond naturally — the system handles tool execution."
  → Current Agent State (full plan as JSON)
  → Phase-specific instructions (INTAKE / EXPLORE / SHORTLIST / COMPARE)
  → SHARED_GUIDELINES (7 rules: concise, listen first, drive forward, use tools correctly, curate top 3, adaptability, stay structured)
```

The `SHARED_GUIDELINES #4` explicitly instructs the LLM **not** to format tool calls as text — it should simply describe its intent and the system handles execution. This prevents the hallucination of XML-style function syntax (`<function=transition_phase(...)>`) that appeared during Build 2.

## 5. Agent Loop Changes: Sprint 2 → Sprint 3

### Sprint 1 Loop (Chatbot)
```
User Message → LLM → updates flat JSON dict → LLM replies
```

### Sprint 2 Loop (Consultant)
```
User Input
  → Build phase-aware prompt + state injection
  → LLM Call 1 with single tool: update_plan
  → If tool called: apply patch → LLM Call 2 for plain text reply (JSON mode)
  → Return: (text_string, updated_plan)
```

### Sprint 3 Loop (Structured Agent — Final)
```
User Input
  → Build prompt (phase-gated instructions + full state JSON)
  → LLM Call 1 with phase-filtered tools (tool_choice="auto")
      → update_plan         → deep-merge patch, preserving unset fields
      → manage_candidates   → upsert/eliminate candidates by name key
      → transition_phase    → validated by _can_transition() server-side
      → generate_mcdm_matrix → populate comparison_matrix rows
  → Execute ALL tool calls in sequence; record tool results
  → [If tools were called] Rebuild prompt with updated state
  → LLM Call 2 (tool_choice="none") — observes results, generates response
  → [If no tools called] Use Call 1's response directly
  → Return: ({ text_reply, comparison_matrix }, updated_plan, new_messages)
```

**Key changes from Sprint 2**:
- 1 tool → 4 tools (phase-gated)
- JSON mode on Call 2 → removed; response is now plain text (no conflicting directives)
- Prescriptive syntax prompts → outcome-focused language
- `VacationPlanPatch.model_json_schema()` → hand-crafted flat schemas
- Dual-call always → conditional dual-call (single call when no tools)
- No phase awareness in loop → full phase state machine with server-side guardrails

## 6. Frontend Changes: Sprint 2 → Sprint 3

The raw JSON debug panel is replaced by a structured **Decision Console** sidebar:

- **Phase Breadcrumb**: Visual progress indicator (Intake → Explore → Shortlist → Compare). Current phase is highlighted; completed phases are dimmed.
- **Constraints Strip**: Displays `origin`, `duration_days`, `budget_range`, `travelers` — visible for debugging and user transparency.
- **Candidate Cards**: Up to 3 active candidates rendered with rank, rationale, and `decision_criteria` badges (amber pills indicating unresolved blockers). Eliminated candidates are not shown.

### New: `ComparisonMatrix.tsx`

Rendered inline below the assistant message that triggers the Compare phase. Displays a trade-off table with `criterion` as rows and active candidate names as columns. Data comes from `comparison_matrix` on the API response — the UI renders it, does not compute it. This is a direct implementation of Server-Driven UI (SDUI).

### Updated: `ChatInterface.tsx`
Accepts `activeCandidateNames` prop (used as column headers in the comparison matrix). Renders `ComparisonMatrix` beneath the relevant assistant message when `comparison_matrix` is non-null on a message.

### Updated: `useAgent.ts`
Attaches `comparison_matrix` from the API response to each assistant `ChatMessage` object so it travels with the message it originated from.

### Updated: `types.ts`
New types: `Phase`, `DestinationCandidate` (with `status`, `rationale`, `pros_cons`, `decision_criteria`), enriched `VacationPlan` (with `phase`, `candidates`, `comparison_matrix`). `ChatResponse` now includes `comparison_matrix`.

## 7. How Real-World Products Do This

### Phase-Gated State Machines
Products like **Expedia's Romi** and **Booking.com's AI assistant** use explicit NLP intent routers that detect when a user has moved from "discovery" (broad queries) to "selection" (comparing specific options). Rather than a single chat prompt, they switch between purpose-built agent configurations for each stage. Our implementation simulates this with a single agent that receives different system prompt instructions per phase.

### The Cart Metaphor
In production travel and commerce systems, candidate tracking is never held in the LLM's context window. It is a first-class database entity — a standard e-commerce cart stored in a persistent backend service (e.g. Redis or Postgres), read by the UI independently of the chat. **The UI reads the cart database, not the chat text.** Our in-memory `SessionManager` + `candidates` array simulates this pattern exactly. The Sprint 4 upgrade path is to move `SessionManager` to Redis.

### Consultant-First Interaction Model
Products that achieve high task-completion rates (Intercom's Fin, Klarna's shopping assistant) use a "diagnose before prescribe" pattern — the agent asks clarifying questions before making proposals. Sprint 3 enforces this explicitly in the Explore phase prompt: ask before suggesting, one destination at a time, track every rejection explicitly.

### Server-Driven UI (SDUI)
Platforms like **Airbnb**, and many fintech products use SDUI patterns where the backend sends JSON payloads that tell the frontend which component to render and with what data — rather than having the LLM emit HTML or Markdown tables. Our `comparison_matrix` response field is a direct implementation: the LLM outputs structured data, and `ComparisonMatrix.tsx` renders it. This eliminates markdown-table hallucination and gives the UI full control over presentation.

### Flat Tool Schemas
Production systems (LangGraph, Anthropic, OpenAI function calling) **always** use explicit flat schemas for tool definitions. Auto-generating schemas from ORM or Pydantic models is an anti-pattern because internal model structure rarely matches the API parser's requirements. The backend handles the "translation" from flat args back into internal models.

### Outcome-Focused Prompting
Real production prompts describe *what the agent should achieve* (e.g., "Ensure all user constraints are recorded") rather than *how to call functions* (e.g., "Call `update_plan(budget=...)`"). Perplexity, Claude API examples, and LangChain documentation all demonstrate this. It is more robust across model families and sizes, and avoids the failure mode where models hallucinate function syntax.

### ReAct Loop
**LangGraph**, **Anthropic's native tool use**, and **OpenAI Assistants API** all implement the same pattern: tool results are always fed back to the LLM before the final response is generated. This is the only way to build a reliable agentic system. Our conditional dual-call pattern is a direct, lightweight implementation of this principle without requiring a framework.

## 11. Sprint 4 Starting Point

Sprint 4 begins from this baseline. The architecture is stable. The deferred items from Sprint 3 (MCDM rendering, observability harness, fallback model improvements) are candidates for inclusion for future sprints. 

Sprint 4's primary focus, per the planning doc, is **Generative UI & Visual Decision Support**: rich destination cards, interactive selection board, and a transition out of the chat interface into a dedicated Trip Dashboard.

Key constraints to respect coming into Sprint 4:
1. **Do not auto-generate tool schemas** from Pydantic models. Always hand-write flat JSON schemas.
2. **Do not remove the second LLM call** in `run_turn()`. The conditional dual-call ReAct loop is correct and must be preserved.
3. **Candidates are upserted by name key**, never replaced as a full array.
4. **The learning notebooks are not part of the build process.**
